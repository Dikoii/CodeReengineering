Change Preventer

Elemen dalam kode yang mencegah perubahan atau penambahan fitur baru dengan mudah. Mereka seringkali terjadi karena desain yang buruk atau ketergantungan yang terlalu kuat antara komponen-komponen dalam sistem. Misalnya, jika suatu kelas memiliki terlalu banyak tanggung jawab atau terlalu banyak metode yang bersifat publik, maka hal ini dapat menjadi change preventer karena akan sulit untuk mengubah atau menambah fungsionalitas tanpa mempengaruhi bagian lain dari sistem.

  • Shotgun Surgery

Divergent Change

Ini terjadi ketika sebuah class atau module sering mengalami perubahan karena berbagai alasan yang tidak terkait. Ini menunjukkan bahwa ada masalah dengan design karena satu bagian dari code yang seharusnya stabil justru harus berubah secara terus-menerus. Ini dapat menyebabkan kesulitan dalam maintenance dan development karena setiap perubahan memerlukan modifikasi besar-besaran.

Treatment

  • Extract Class
  • Extract Superclass
  • Extract Subclass

Contoh, jika kita memiliki class `Employee` yang bertanggung jawab untuk mengelola data karyawan, namun tiba-tiba kita ingin menambahkan fungsi untuk menghitung gaji, maka class ini mengalami divergent change. Solusinya, Kita dapat memisahkan tanggung jawab ini dengan membuat class `EmployeeDetails` yang khusus untuk mengelola data detail karyawan seperti gaji dan usia. Hal ini menghilangkan divergent change dengan memisahkan tanggung jawab yang berbeda ke dalam class-class yang terpisah.

Before

                // Order.java:
class Employee {
    private String name;
    private double salary;
    private int age;

    // Methods related to employee management
    public void setName(String name) {
        this.name = name;
    }

    public void setSalary(double salary) {
        this.salary = salary;
    }

    public void setAge(int age) {
        this.age = age;
    }

    // Divergent change: New method added for salary calculation
    public double calculateSalary() {
        // Calculation logic
    }
}

After

                // Order.java:
class Employee {
    private String name;
    private EmployeeDetails details;

    public void setName(String name) {
        this.name = name;
    }

    public void setDetails(EmployeeDetails details) {
        this.details = details;
    }
}

class EmployeeDetails {
    private double salary;
    private int age;

    public void setSalary(double salary) {
        this.salary = salary;
    }

    public void setAge(int age) {
        this.age = age;
    }

    public double calculateSalary() {
        // Calculation logic
    }
}

Shotgun Surgery

Merujuk pada situasi di mana setiap kali Anda ingin membuat perubahan kecil, Anda harus memodifikasi banyak bagian kode yang berbeda. Ini adalah tanda bahwa desain Anda terlalu terfragmentasi dan tidak teratur, karena perubahan yang semestinya sederhana justru memerlukan "sayatan" besar-besaran ke berbagai bagian kode.

Treatment

  • Move Method
  • Move Field
  • Inline Class

Contoh, kita memiliki banyak class yang memiliki method untuk memformat tanggal secara terpisah, dan jika ingin mengubah method pemformatan tanggal tersebut kita harus mengubahnya satu-persatu ini termasuk ke dalam "Shotgun Surgery". Solusinya, kita dapat memindahkan logika pemformatan tanggal ke dalam class `DateUtils`, yang membuatnya lebih mudah diubah dan menghindari perubahan yang merata di berbagai class terkait.

Before

            // Order.java:
class Order {
    private Date orderDate;

    // Method to format date for display
    public String formatDate() {
        // Formatting logic
    }
}

class Invoice {
    private Date invoiceDate;

    // Method to format date for display
    public String formatDate() {
        // Formatting logic
    }
}

After

            // Order.java:
class DateUtils {
    public static String formatDate(Date date) {
        // Formatting logic
    }
}

class Order {
    private Date orderDate;

    public String getOrderDateFormatted() {
        return DateUtils.formatDate(orderDate);
    }
}

class Invoice {
    private Date invoiceDate;

    public String getInvoiceDateFormatted() {
        return DateUtils.formatDate(invoiceDate);
    }
}

Parallel Inheritance Hierarchies

Terjadi ketika perubahan di satu hierarki kelas harus diikuti oleh perubahan di hierarki kelas lainnya. Ini terjadi ketika kelas-kelas saling tergantung secara eksklusif di antara hierarki-hierarki tersebut, yang dapat menyebabkan kesulitan dalam memodifikasi atau menambah fungsionalitas tanpa memengaruhi hierarki yang lain.

Treatment

  • Move Method
  • Move Field

Contoh, jika kita memiliki hierarki class untuk produk dan hierarki class untuk kategori produk, dan setiap kali kita menambahkan kategori baru, kita harus mengubah class-class di kedua hierarki ini. Solusinya, kita dapat kita menghilangkan hierarki class untuk kategori produk dan menggantinya dengan satu class `Category` yang terintegrasi dengan class `Product`, mengurangi ketergantungan dan perubahan yang saling terkait.

Before

            // Order.java:
// Product hierarchy
class Product {
    // Product attributes
}

class Book extends Product {
    // Book-specific attributes
}

// Category hierarchy
class Category {
    // Category attributes
}

class BookCategory extends Category {
    // Book category-specific attributes
}

After

            // Order.java:
class Product {
    // Product attributes
    private Category category;

    public void setCategory(Category category) {
        this.category = category;
    }
}

class Category {
    // Category attributes
}
